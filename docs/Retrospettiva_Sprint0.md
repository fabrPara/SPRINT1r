# Retrospettiva Sprint 0 - Gruppo "Barto"
> **Obiettivo:** Analisi del processo e miglioramento continuo (12° Principio Agile).

| Categoria | Osservazione | Teoria | Azione Correttiva |
| :--- | :--- | :--- | :--- |
| **EVITARE** | **Gestione Traccia:** Poca attenzione dell'intero gruppo nelle piccolezze della traccia assegnata dal professore. | La troppa "leggerezza" dello Sprint 0 ha portato alla conseguenza di sottovalutare i dettagli della traccia. | Maggiore attenzione e riletture multiple della traccia principale. |
| | **Comunicazione:** Poca interazione nel gruppo riguardo ai problemi dello stesso. | Sottovalutiamo la difficoltà di alcuni compiti e tendiamo a isolarci senza discuterne con il team. | Non sottovaluteremo nessun compito e affronteremo ogni lavoro con la stessa voglia e determinazione. |
| | **Autonomia:** Poco lavoro individuale e autonomo da parte di alcuni membri del gruppo. | Spaventati dalla difficoltà di certi compiti, pensiamo di dover dipendere in maniera forzata dal gruppo invece di contare sulle proprie capacità. | Conteremo più su noi stessi, cercando di eseguire al meglio il compito in maniera individuale. |
| | **Workflow PR:** Mancata indicazione dei Reviewer e chiusura delle Pull Request senza la revisione di più persone. | La troppa fretta nel completare i nostri compiti ci ha portato a saltare questi passaggi di controllo. | Provvederemo a inserire più Reviewer in ogni Pull Request. |
| | **Gestione Git:** Chiusura ritardata dei branch e push diretti sul ramo principale. | La poca familiarità iniziale con gli strumenti di GitHub e la fretta hanno generato confusione nella gestione dei rami. | Controlleremo che ogni branch venga chiuso a lavoro finito ed effettueremo controlli affinché il main non venga modificato. |
| | **Punto Dolente:** Gestione ansiosa dei tempi e fretta eccessiva nel finire i compiti. | Tendenza a voler finire in maniera frettolosa per paura di non rispettare le scadenze stabilite. | Lavoreremo con più calma, utilizzando correttamente tutto il tempo messo a disposizione dal professore. |
| | | | |
| **CONTINUARE** | **Daily Meeting:** Aggiornamento continuo su ciò che il gruppo sta eseguendo. | Effettuare incontri quotidiani di 10-15 minuti in cui il gruppo riflette sugli obiettivi giornalieri da imporsi. | Mantenere i Daily Meeting costanti sulla piattaforma Discord. |
| | **Qualità Commit:** Messaggi di commit esaustivi ma allo stesso tempo concisi. | Continuare a scrivere messaggi di salvataggio chiari aiuta a comprendere meglio l'evoluzione del codice. | Controlleremo che i commit vengano scritti sempre in questo modo preciso. |
| | **Puntualità:** Terminare l'intero lavoro da eseguire nei tempi previsti. | Nonostante la fretta, il lavoro è stato eseguito in maniera ottima e rispettando le scadenze. | Continueremo a lavorare rispettando le date di consegna nel nostro interesse. |
| | **Clima di Team:** Collaborazione tra i membri e ambiente lavorativo sereno senza sfide negative. | L'aiuto reciproco e il supporto in videochiamata hanno permesso a ogni membro di lavorare correttamente. | Manterremo questo ambiente sereno, tranquillo e di supporto reciproco anche nei prossimi sprint. |
| | | | |
| **INTRODURRE** | **Analisi Errori:** Revisione sistematica degli sbagli o dei potenziali errori commessi il giorno precedente. | Prevediamo che discutere i difetti e gli errori passati possa prevenire danni futuri al progetto. | Inseriremo l'analisi dei difetti e degli errori potenziali all'interno dei nostri incontri quotidiani. |
| | **Indipendenza:** Sviluppo di un metodo di lavoro più autonomo senza dipendere dagli altri. | Lavorare in modo più indipendente darà una spinta in più alla qualità del proprio lavoro. | Favoriremo il lavoro individuale senza sminuire l'importanza del confronto con la squadra. |
| | **Personal Board:** Uso di Scrum Board individuali (anche cartacee) per ogni membro del gruppo. | Annotare e seguire i propri compiti individualmente aiuta a gestire meglio le attività da portare al termine. | Ogni membro cercherà di gestire una propria tabella di supporto per prendere nota del lavoro svolto. |
| | **Peer Review:** Revisione collettiva di ogni Issue terminata e approvazione esplicita del team. | Il team dovrà revisionare accuratamente ogni compito terminato per non riscontrare problemi precedenti. | Ci impegneremo a controllare collettivamente il lavoro di ogni membro prima di considerarlo concluso. |
